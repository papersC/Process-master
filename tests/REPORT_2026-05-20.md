### Medium

**M-1. `Authenticated_Post_Without_Antiforgery_IsRejected` is misnamed and tests the wrong thing.**
- **Where:** `ESEMS.Tests/Integration/AuthorizationSweepTests.cs:130-153`
- **Repro:** `dotnet test --filter "FullyQualifiedName~Authenticated_Post_Without_Antiforgery"` — fails 3 cases (`/Improvements/CreateFromWizard`, `/Incidents/Create`, `/EnterpriseRisks/Create`) with `Expected: Not OK; Actual: OK`.
- **Root cause:** `EsemsTestFactory` registers `NoOpAntiforgery` (lines 61–62 + 78–96). Real antiforgery only fires when the caller adds the `X-Test-Validate-Antiforgery: true` header — which this test does not. So the request never gets rejected for CSRF; the test is actually verifying "endpoint rejects a partial POST body without form data". Three endpoints return 200 (View with model errors) instead of 302/400, so the assertion `Assert.NotEqual(OK, …)` fails. Production CSRF protection is real and unaffected.
- **Suggested fix:** Rename the test to `Authenticated_Post_With_PartialBody_Rejects` and either (a) add `X-Test-Validate-Antiforgery: true` to genuinely exercise CSRF, or (b) adjust the assertion to accept 200 with ModelState errors as a valid rejection path.
- **Owner:** test maintainer (not a deployment blocker).

**M-2. Coverlet under-reports Razor coverage.**
- **Where:** `artifacts/test-results/d44376ba-bae3-4147-b78c-4687542e02fd/coverage.cobertura.xml`
- **Symptom:** every `Views/**/*.cshtml` shows `line-rate="0"` despite being rendered by integration tests.
- **Root cause:** the Razor-compiled `AspNetCoreGeneratedDocument` classes aren't instrumented the same way regular C# is. Coverlet sees them as 0%.
- **Suggested fix:** filter coverage to `[ESEMS.Web]ESEMS.Web.Controllers.*;[ESEMS.Web]ESEMS.Web.Services.*;[ESEMS.Web]ESEMS.Web.Helpers.*;[ESEMS.Web]ESEMS.Web.Security.*` via a `runsettings` file. Or switch to `dotnet-coverage`. Or accept that Razor coverage is exercised by E2E (Tier 2) and not by unit/integration.
- **Owner:** infra (cosmetic; coverage report only).

**M-3. Asset DataOwner round-trip not tested via POST.**

### Tier 2 — additional findings

**M-4. Playwright .NET 1.49 needs Chromium build 1148, but the local cache only had builds 1208/1217.**
- **Symptom:** the first full-suite run completed in 63 ms (impossible for 31 real browser tests). The fixture's catch block swallowed the browser-launch failure and silently flipped `ServerReachable = false`, so every test returned at line 1.
- **Fix this session:** `pwsh ESEMS.E2E/bin/Debug/net9.0/playwright.ps1 install chromium` downloaded the correct build.
- **Suggested durable fix:** add a CI step that runs `playwright install --with-deps chromium` before any `dotnet test ESEMS.E2E`, and make the fixture *throw* (not silently skip) when the browser is missing AND an env var like `ESEMS_E2E_REQUIRE_BROWSER=1` is set.
- **Severity: Medium.** Risk: silent skips look like passes in CI.

**M-5. E2E flakiness when the full suite runs as a single batch.**
- 5 tests pass alone but fail together: `InlineOnclickPolyfillTests.Polyfill_FiresExpandAll_OnProcessHierarchy`, `AuthenticatedSmoke.ImprovementsList_RendersWithoutConsoleErrors`, `PrintAndUploadTests.PrintStylesheet_HidesChrome_OnListPage`, `LoginAndCultureSmoke.ProtectedRoute_Without_Auth_Redirects_To_Login`, and one en-US case of `FCC003_Breadcrumb_RendersOnServiceCategoriesIndex`.
- **Root cause (likely):** all tests run against a single `dotnet run` instance and share the same Chromium browser process; locator timeouts (default 5 s) trip when the server is contended.
- **Suggested fix:** bump `Assertions.Expect(…).ToBeVisibleAsync(new() { Timeout = 15_000 })` across the existing smoke tests, OR collection-fixture-scope the browser context so tests don't fight each other, OR run with `-parallel none`.
- **Severity: Medium.** Risk: dev gets used to ignoring "flaky" E2E failures and misses real ones.

**M-6. PII masking E2E test depends on seeded feedback having an email field.**
- `FOP005_FeedbackDetails_MasksPIIByDefault` walks `/CustomerFeedback` and picks the first record with `data-pii=email` in its Details markup. If the seed data ever changes to omit email, the test silently returns (no assertion failure, looks like pass).
- **Suggested fix:** make the test seed its own feedback row via the API/DbContext in `InitializeAsync`, then read it back. Same pattern as `Regression_2026_05_20_Tests.cs` in the integration project.
- **Severity: Low.** Risk: test silently passes when it has nothing to assert.

### New findings (Tier 3)

**M-7. CSP `style-src` allows `'unsafe-inline'`.**
- **Where:** `ESEMS.Web/Middleware/SecurityHeadersMiddleware.cs` (CSP emitter).
- **Evidence:** `LiveServerSecuritySmoke.CSP_StyleSrc_DoesNot_Allow_UnsafeInline` fails — actual CSP `style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net https://cdnjs.cloudflare.com https://fonts.googleapis.com https://cdn.datatables.net https://unpkg.com https://cdn.tailwindcss.com;`.
- **Severity: Medium.** Inline-style injection via Razor-rendered attributes can be exploited by CSS-only data-exfiltration attacks (e.g., extracting CSRF tokens through attribute selectors + `background-image: url(…)`).
- **Suggested fix:** nonce-ify the few remaining `style="…"` attributes (audit count: high in `_Layout.cshtml` and the new `_RelatedChangeRequests.cshtml` partial), then drop `'unsafe-inline'` from `style-src`.

**M-8. Response leaks server software identity.**
- **Where:** Kestrel default; `SecurityHeadersMiddleware` doesn't strip the `Server` header.
- **Evidence:** `LiveServerSecuritySmoke.Response_DoesNot_Leak_ServerSoftwareHeader` fails — `Server: Kestrel`.
- **Severity: Low.** Reveals the platform to attackers fingerprinting for known-CVE exploits, but doesn't directly expose data. In production behind IIS the header value differs (`Microsoft-IIS/10`), which is similarly leaky.
- **Suggested fix:** `app.Use(async (ctx, next) => { ctx.Response.OnStarting(() => { ctx.Response.Headers.Remove("Server"); return Task.CompletedTask; }); await next(); });` or configure Kestrel's `AddServerHeader = false`.

**M-9. CSP whitelists 6 third-party CDNs in `script-src`.**
- `cdn.jsdelivr.net`, `cdnjs.cloudflare.com`, `unpkg.com`, `cdn.datatables.net`, `cdn.tailwindcss.com`, plus `cdnjs.cloudflare.com` listed twice.
- **Severity: Low.** Supply-chain risk: a compromised CDN can serve attacker-controlled JS that the nonce can't catch. Mitigations: self-host the critical libraries (Bootstrap, Tailwind, DataTables, jQuery, Lucide) and add SRI hashes (`integrity="sha384-…"`).
- Cross-references the static audit finding **F-CC-LO-3** (Tailwind CDN production warning) — same root cause.

**M-10. Color-contrast on breadcrumb gray-400 across 18/20 pages.**
- **Where:** `ESEMS.Web/Views/Shared/_Breadcrumb.cshtml`. The `text-gray-400` Tailwind class renders `#9ca3af` on white, contrast ratio ~2.8:1 against WCAG 2.1 AA's 4.5:1 minimum.
- **Severity: Medium.** Affects every authenticated page. Users with low vision can't read breadcrumb middle segments.
- **Suggested fix:** swap `text-gray-400` → `text-gray-600` (#4b5563, contrast ~7.2:1).

**M-11. `scrollable-region-focusable` on the home dashboard activity feed.**
- `.dash-activity-body` is a vertically-scrollable region without `tabindex="0"` or a focusable child, so keyboard-only users can't scroll into it.
- **Severity: Medium** (keyboard accessibility = WCAG 2.1.1 hard requirement, level A).
- **Suggested fix:** add `tabindex="0"` and `role="region"` + `aria-label="Recent activity"` to the wrapper.

**M-12. Three unlabeled inputs on SettingsHub > General tab.**
- Inputs for `L1.png`, `MBRHE`, language code `en` (settings keys 020538A6…, 58F21457…, 6CAB4C7C…) have no `<label>` or `aria-label`.
- **Severity: Medium** (WCAG 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions).
- **Suggested fix:** wire each settings row to its key label via `aria-labelledby` referencing the existing key cell.

**M-13. `/Processes` `<select>` filter lacks accessible name.**
- A DataTables-driven filter `<select>` on the Processes list page has no associated label.

### Persistent false positive

**M-15 axe complaint persists despite DevTools confirming `color: rgb(255, 255, 255)` on white-painted nav labels over `rgb(42, 48, 66)` sidebar (contrast 12.5:1).** axe-core 4.10 has a documented limitation: when an element has `background-color: rgba(0, 0, 0, 0)` (transparent) AND the next non-transparent ancestor is `display: none` (e.g. a collapsed submenu container), the contrast walker fails open and reports the violation. Every page hits this because every authenticated layout includes the collapsed-by-default submenus.

**Mitigation options for follow-up:**
1. Set a non-transparent `background-color` on the submenu container even when `display: none` (Tailwind `bg-primary-700` does this — but Tailwind CDN may not be applying it at the moment the test runs).
2. Tag the affected `<span>`s with `aria-hidden="true"` only when the submenu is collapsed — but that breaks visual labeling.
3. Add an axe-core suppression rule for `#service-menu .nav-label` and similar (acknowledge the FP).
4. Replace the Tailwind play CDN with a built Tailwind stylesheet so the BG color actually paints in the inspected state.

### M-16 hard-coded cascade

`a { text-decoration: none !important }` somewhere in the Skote / Bootstrap stack beats every `!important`-flagged rule I tried, including an inline `style=` attribute. The next move is to grep the vendor CSS for the rule and either override it at the global site.css level or refactor the link out of `dash-activity-head` so it's no longer "in a text block" (axe rule only fires when the link sits adjacent to body text). Deferred — not deployment-blocking.
